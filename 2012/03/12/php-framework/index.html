何为PHP框架

上次同事说了句话，感慨挺深的，他说：

整天看PHP视频啊博客啊，整天在 PHPCMS 里面打转啊可就是不知道框架是什么意思！

我估计没写过框架的人是不知道怎么才是框架的，毕竟很多事情得自己亲身经历一番才能知道其中奥妙。

事实上框架的概念很大又很小，比如我之前的 MagicASP(ASP下的路由分发框架)，其实只是实现了个路由分发的功能，但是我敢说这就是框架，为什么呢？首先咱们来看一下框架的意思。

Wiki 百科:软件框架中

软件框架（Software framework），通常指的是为了实现某个业界标准或完成特定基本任务的软件组件规范，也指为了实现某个软件组件规范时，提供规范所要求之基础功能的软件产品。
框架的功能类似于基础设施，与具体的软件应用无关，但是提供并实现最为基础的软件架构和体系。软件开发者通常依据特定的框架实现更为复杂的商业运用和业务逻辑。这样的软件应用可以在支持同一种框架的软件系统中运行。

简而言之，框架就是制定一套规范或者规则（思想），大家（程序员）在该规范或者规则（思想）下工作。或者说就是使用别人搭好的舞台，你来做表演。

这样就明白了其实框架就是一个部好景的舞台，你只需要在里面表演就行。

事实上虽然这样的解释很轻巧，很符合人的常理，但是它还忘了一个至关要紧的点：框架也是工具，好工具的特征就是能给你省下大把的时间。所以一个框架如果不能给你带来时间上的优势那么你还不如不用它。

这是选择框架上最至关重要的一点，当然还有其他点，比如：

  1. 框架文档的齐全[节省上手时间]
  2. 代码编写规范[可以规范代码，提升代码质量]
  3. 主要的设计思想[不同的框架有不同的适合场景]
  4. 强大的社区[有问题靠大家]
  5. 使用人数[节省时间]
  6. 文件大小[文件大小大的框架提供更多的特性，同时你也得为磁盘跟额外的负荷买单]
  7. 等等等，自己看着办

其实对于一个像我这样的菜鸟 PHPer，很多时候利用框架就是等于浪费自己的生命，最开始的时候学习 Yii [名字取为易，实在是糟蹋了汉字的博大精深]，光学习它的事件驱动规则以及ORM就浪费了很多时间，我承认它是将我从SQL语句中释放出来了，但是它有把我带入另外一个坑了。[确实用 ORM 用的很蛋疼，还有蛋疼的事件驱动，代码写的很分散，独自开发的时候全浪费在拖动滚轮上了。但无论如何它还是很优秀的。只是对于我一个个人开发者，它太庞大。]

后来换了 CodeIgniter 。用了 CI 后，终于感觉好点了，很适合我。该有的都有，文档也好。

事实上我感觉，对于一个PHPer的框架，写写小项目的话，至少应该实现以下的功能：

  1. 路由分发[ 是必须的，标配的]
  2. 数据库操作类[工厂模式，MySQL 什么的是必须的]
  3. Utility [常用工具集，很多人误解了 Utility 跟 Framework 的区别，两者都可以节省你的时间，但是最大的不同之处在于，你一但引入了框架，那么就必须照着框架的模式进行开发，而你引入了 Utility 却可以不必管它。所以我为什么说我的 MagicASP 是框架，而 EasyASP 不是框架的原因了。]
  4. 以及一些必须的安全细节跟设置细节操作[比如转义输入，设置时区]

虽然 PHP 中每种框架都有自己的特点，但是大体难以逃过以上几点的基本细节。以上的细节无疑是非常好实现的，所以 PHP 框架的种类可以说是非常的多，我感觉语言里面框架最多的就是 PHP 了，随手拈来就是一个。当然质量也就参差不齐了。

当然好的框架可能会实现更多的细节,比如:异常处理，本地化处理，代码生成，命令行支持，模版引擎 等等诸多繁琐的细节，基本上这样的框架是很恐怖的，一般都被用来当教科书或者参考书的，比如 Zend 框架。

至于 MVC 呢，PHPer 口口声声说的 MVC 呢，哪去了，我感觉 MVC 不是必须的，你写代码的时候记得分3个文件夹 Module, View, Controller，基本上就实现了 PHP 中的 MVC 。

以上这段文字是当年的戏言，现在想想太过偏激，事实上选择 DAO 或则 Model 层都是可以的，但是不可将过多的业务逻辑放在 Controller , 建议封装出一层服务层给 Controller 调用，而 View 则仅仅是渲染数据的。事实上就像是在 Web 端谈 MVVM 一样，在 PHP 端的 MVC 事实上已经大幅缩水。

使用框架更多的是好处，但是也有敝处，比如绑定开发者的习惯，效率问题，额外的学习时间等等。但是这相对有使用框架的好处来讲是可以忽略的。

PHP框架界的繁荣并非好事，满眼的框架带给PHPer更多的困惑。没有一个犹如 Ruby 中 Ruby On Rails 重量级的框架，无疑是 PHPer 的硬伤。
