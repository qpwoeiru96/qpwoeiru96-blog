新的开始

博客又迁移了，从 qpwoeiru96.me 到 sou.la/blog 再到现在的 blog.sou.la, 从 WordPress 到 Typecho 再到 Hexo，越来越轻的程序，到了 Hexo 甚至已经没有了后台，有的只是命令行，而我也越来越享受写作这样的乐趣。

抛开很多很水的文章不讲，说实话没有几篇是自己“满意”的，其实我已经删除或者隐藏了很多我自己当时觉得不错的但是没过多久又觉得很糟糕或者不值得一提的文章。

程序员大多都有 “reinvent the wheel（重复造轮子）” 的毛病，究其原因无非就是 “虚荣心” 作祟，用别人的不好意思呗。我觉着这是好事但同时也是一件坏事，特别是在现在互联网快速发展的时代，效率决定着很多环节，道哥的这篇 《论“重复造轮子”》就举了一个生动的例子。

每当我写文章之前，总是先 Google 一下这个类型有没有写过，别人到底写的好不好，以防止无意识的重复造轮子 。然而每每总是乘兴而来，败兴而归。

第一：人家确实写的很好，该讲的都该讲到了，即使偏门点也有其他文章补充。

第二：该死的 HTTP 协议，你会了解到你太多你之前未了解的东西了。

第三：等我完整的看完我 Google 到的文章之后，我发现时间早已经过去两三个小时了。

第四：就像描述问题总是很简单一样，而解决问题则需要太多太多的上下文以及知识了。

但其实我们又是重复造轮子的受益者，阮一峰 每每写的文章我都去拜读，其实阮一峰很少创造真正意义上自创的文章，更多的是将知识浓缩一下以通俗易懂的方式讲述出来。我不知道这是否属于重复造轮子？但我欣赏这样的重复造轮子。

说到底我们反对的不是重复造轮子，反对的而是粗制滥造的轮子。最近在运用 .Net C# 进行编程，特别深有体会， Google 很多问题只是给出了解决方式，而没有给出问题发生的真正原因。其实就是我说的第四条一样：解决问题需要太多太多的上下文及知识了。而了解这些所用时间的挫败感明显超过了解决问题本身所带来的成就感。

以上困扰着我长达几年的写博生涯，直到最近才悟出，就像编程一样，我也可以通过一些东西来弥补我的困惑。

  1. @todo。这个知识\问题你现在不了解，不代表你未来不了解。
  2. @see link。加个提示，文章总有侧重点，未覆盖的不一定不了解只是篇幅有限而已。
  3. include。优秀的文章的引用往往能减少很多时间。就像使用了开源软件一样。
  4. buffer。缓冲是很有必要的，长篇大论往往容易使自己打退堂鼓，也容易使读者认为你太监了。

希望使用了以上的法则，能使我在 2014 下半年 创作出更多的优秀文章。
